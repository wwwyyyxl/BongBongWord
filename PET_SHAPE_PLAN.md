# BongBongWord Pet 形态方案：纯血鸿蒙 6.0+ 学习伙伴型 Lexi

## 1. 结论

Lexi 是 BongBongWord 的学习伙伴，不是需要用户额外照顾的电子宠物。

它的核心价值是把用户的学习行为变成可感知的陪伴、反馈和成长：

```text
学习行为 -> Lexi 情绪反馈 -> 经验 / 阶段成长 -> 鼓励继续学习
```

BongBongWord 以纯血 HarmonyOS 6.0+ 打造，Pet 能力优先使用 ArkTS、ArkUI 和 HarmonyOS 原生能力实现，不按 Web、Android 或跨端妥协方案设计。

## 2. 方案对比

| 方案 | 说明 | 取舍 |
| --- | --- | --- |
| 轻陪伴型学习伙伴 | 学习行为触发情绪、经验、成长反馈 | 最贴合背单词主目标，选用 |
| 强养成型电子宠物 | 引入饥饿、清洁、生病、喂养等完整宠物系统 | 容易让用户额外照顾宠物，暂不选 |
| 纯装饰型角色 | 只展示形象和点击反馈 | 成本低，但学习激励弱，暂不选 |

最终选择：轻陪伴型学习伙伴。

## 3. V1 产品边界

V1 要做的能力：

- 背词完成后，Lexi 给出开心或鼓励反馈。
- 复习完成后，Lexi 增加经验值或亲密度。
- 答对时，Lexi 表现为开心、认可、连击鼓励。
- 答错后订正时，Lexi 表现为鼓励，而不是惩罚。
- 连续打卡时，Lexi 触发阶段成长或特殊祝贺。
- 阶段升级时，Lexi 展示更明显的成长反馈。

V1 明确不做：

- 不做饥饿值。
- 不做清洁值。
- 不做生病状态。
- 不做强制喂养。
- 不做离线掉状态。
- 不做用户未照顾宠物后的惩罚。
- 不做脱离学习主流程的独立养成游戏。

## 4. V1 状态模型

Pet 状态只服务学习反馈，不表达生存照顾压力。

```text
mood          当前情绪，例如 idle / happy / encourage / sleep / levelUp
level         Lexi 当前等级
exp           当前经验值
intimacy      亲密度，用于表达长期陪伴感
lastStudyTime 最近一次学习行为时间
```

状态含义：

- `mood` 用于驱动 ArkUI 表现层，例如表情、气泡、轻动画。
- `level` 和 `exp` 来自学习行为，不来自喂养或照顾。
- `intimacy` 表达持续学习陪伴感，不作为惩罚系统。
- `lastStudyTime` 用于展示欢迎回来、继续学习等反馈，不用于离线惩罚。

## 5. 学习事件映射

| 学习事件 | Lexi 反馈 | 成长变化 |
| --- | --- | --- |
| 完成一组背词 | 开心 / 鼓励 | 增加经验值 |
| 完成一组复习 | 认可 / 稳定陪伴 | 增加经验值和亲密度 |
| 答对题目 | 开心 / 连击鼓励 | 小幅增加经验值 |
| 答错后订正 | 鼓励 / 重新振作 | 小幅增加亲密度 |
| 连续打卡 | 祝贺 / 特殊反馈 | 增加阶段进度 |
| 达到经验阈值 | 升级 / 进化反馈 | 提升等级 |

## 6. 工程边界

`features/pet` 作为独立 HarmonyOS shared 模块承载 Pet 能力。

页面侧只触发学习事件，不直接散落宠物状态逻辑。

```text
学习页面 / 复习页面 / 打卡页面
  -> 触发学习事件
  -> features/pet 计算状态变化
  -> ArkUI 组件展示 Lexi 反馈
```

后续代码落地时遵循：

- ArkTS 静态严谨性优先，不套 JavaScript / TypeScript 宽泛写法。
- ArkUI 组件只负责展示和交互，不把学习业务堆进 `build()`。
- Pet 状态变化集中在 Pet 模块内部，页面只传递学习事件。
- 资源和组件由 Pet 模块统一管理，页面不散落图片路径和状态枚举。

## 7. 视觉与动效路线

当前 3D 指的是视觉风格和角色资产方向，不等于必须引入真实 3D 渲染。

表现层按以下优先级逐步升级：

```text
静态 3D 风格图片
  -> ArkUI 轻动画 / ImageAnimator
  -> Lottie
  -> 真实 3D 模型渲染
```

只有拿到真实 `.glb` / `.gltf` 模型，并确认 HarmonyOS 6.0+ 原生渲染链路、性能和包体都可接受后，才考虑真实 3D 方案。

## 8. 后续需求判断准则

任何新的 Pet 需求，都先问三个问题：

1. 它是否增强了背词、复习、练习或连续学习反馈？
2. 它是否让用户产生额外照顾负担？
3. 它是否符合 HarmonyOS 6.0+ 原生 ArkTS / ArkUI 实现边界？

默认选择能增强学习反馈、不会增加照顾压力、最贴近鸿蒙原生能力的方案。

## 9. 当前默认假设

- 当前阶段只沉淀形态方案，不实现 UI。
- 当前阶段不接学习业务事件。
- 当前阶段不引入 3D 引擎。
- 后续可以迭代玩法，但不能破坏“学习行为可视化伙伴”这条主线。
- 若后续讨论 Kotlin 版本，只做业务规则参考，不反向牵制当前纯血 HarmonyOS 实现。
